API Documentation

A look into the University Faculty API



Starting point

The API can be accessed through this given starting URI http://comp3227.ecs.soton.ac.uk/modules/15, which links to a resource of a module in the API. From here we can explore the API and the rest of the endpoints located inside it.

General URI Format

Each URI has a commonly shared structure which is made of multiple sections:

  1. Each URI specifies the http protocol at the beginning of the URI as 'http://', which is the protocol used to access the resources in the API.
  2. The domain of the URI is comp3227.ecs.soton.ac.uk, which identifies the host origin server for the API
  3. The path comes next after the general root URI 'http://comp3227.ecs.soton.ac.uk', and is designated through a series of '/' characters and keywords e.g /modules/15
  4. Finally a resource identifier can be used at the end to locate a specific resource e.g modules/15 has the identifier '15'

For the example URI http://comp3227.ecs.soton.ac.uk/modules/15, its individual parts have been colour coded. Here we can see it specifies the 'http' protocol, identifies the origin server at 'comp3227.ecs.soton.ac.uk' and follows a path of /modules/ and specifies module resource located at identifier '15'.

  • If the value in the {Module id} is not recognised by the API, a 404 Not Found Response is returned as well as a 'Not Found' description in the response body.

PUT Method

  • Using the PUT method allows you to replace fields of the representations of the module with new values, giving a 200 OK response if the call is successful.

  • In the body of the request, you can include fields with the values you wish to change them to, like in the request below. The format of the body is further detailed in the Options section.

A request to modify a module
A successful response

  • If incorrect authentication is provided, a 403 Forbidden response is returned. If no authentication header is included a 401 Unauthorised code is returned. Additionally, if the request body cannot be parsed correctly or specified fields aren't filled in the request body, a 400 Bad request is returned and the URI remains unchanged. The response body also replies with any fields that must be included in the request body.

  • If any values are out of range e.g a year not being less than or equal to five, a 400 Bad Request response is given.

A response to a bad PUT request

PATCH Method

  • Using the patch method allows you to replace values in the representation with new values.

  • The server responds with a 200 OK response code and updates the URI if the call is successful.

PATCH does not require any fields to be present in the request body, but it still retains the same constraints to the fields that are included as PUT (Which is given in the body of an OPTIONS request), such as the semester field data having an integer value between 1 and 2. If these constraints are not met, the server will respond with a 400 Bad request code, and no data will be changed in the resource. The response body will also include an error message, for example saying "A valid integer is required" if the semester field is entered as a string "two".

DELETE Method

Using the delete method allows you to send a request to delete one of the modules listed. If the server accepts your authentication, the module is removed from the server and a 203 No Content Message is returned. The server will keep processing the delete method until it times out.


If no authentication header is added to the delete request, a 401 Unauthorised Response code is returned and the module does not get deleted. If you try to delete a module without the correct Authentication, a 403 Forbidden response code is returned and the module does not get deleted.

OPTIONS Method

Using the OPTIONS method returns the formats of the representations stored in the API Server, which are in JSON, XML and CSV, and a 200 OK response if the call is successful. It also gives information about how the body of a data-changing method like PUT or PATCH is parsed, showing that the message can be parsed in JSON, XML, or as CSV text.


If the value in the {Module id} is not recognised by the API, a 404 Not Found Response is given.

Options Response with valid Authorisation

If correct authorisation is provided the options body also includes extra information for a put command that adds some constraints to what can be included in the body of a PUT request. For example, for the url field it is set so it can only be read and not modified, so the client cannot choose the location of where the module is stored. It also gives the types allowed, such as only integers for the year or any value for the url. Each field has a descriptive label, and some fields are required to be inside the PUT request body, such as name, code and faculty. Some fields have restraints on the values they take, for instance with the semester field, it can only take integers between 1 and 2. If these constraints are not fulfilled in the PUT request, it will not be successful and return a 400 Bad Request response.

GET Method

Using the get method returns a representation of the module list resource and a 200 OK response if the call is successful, as well as the servers headers that they choose to show in their response.

The representation is given as a list of modules spanning across every valid endpoint http://comp3227.ecs.soton.ac.uk/modules/{module_id}, containing information about the module name, url and code. not all information about the individual module is given, e.g the year and semester it takes place in and the faculty and courses it belongs to.

Representation Format

The formats allowed for the returned representation are in JSON, XML and a CSV Text table, which can be requested for through an Allow Header in the request headers. The default representation given by the server is in JSON format. If the client requests a representation not stored by the server, e.g ecmascript, then the server responds with a 406 Not Acceptable Response code, and details in the response body that the server could not satisfy the request Accept Header.

POST Method

The POST method allows the client to send a request to create a new URI in the web server. For the request to be successful, an Authorisation header must be added to the POST request including A string with a Token Value in, formatted as "Token: {Token_Val}". If no authorisation is provided, the server will Respond with a 401 Unauthorised Code and include a message in the body that says authorisation was not provided. If incorrect authorisation is provided, the server will respond with a 401 Unauthorised code and say that the Token is Invalid.

If the POST method is accepted by the server, it will respond with a 201 Created Response Code, and will display the new representation under a new URI, which is referenced by the url field, in the body in JSON XML or CSV. The Response also includes a location header which tells the client the endpoint that the module resource is stored in. The representation of this module can then be obtained through the endpoint that the server specifies.

Options Body without Authorisation

The body shows that the name of the resource is labelled as a list of modules with no description for the description field. Additionally, it shows that the formats the representation can take are in JSON, XML and CSV in the renders sections and that messages in the client request body are parsed in JSON XML and CSV as well.

Options Body with Valid Authorisation

This adds extra information about the settings on the actions that the POST method is allowed to take. For example it requires a POST request body to include the field "name", limit the values for the "year" field to between 1 and 5 and limit the string size for the "code" field. It can also set which fields are read_only, meaning they would not replace the field data for that specified field.
